 SECTION 1 — VARIABLES, TYPES & TYPE INFERENCE
===============================================================================

NOTES
-----
- Dart is STATICALLY TYPED with TYPE INFERENCE. You can write `var x = 5` and
  the compiler infers `int`. Once inferred, the type is fixed.
- Declaration keywords:
    var      -> mutable, type inferred and then fixed.
    final    -> single assignment, value set at RUNTIME, then immutable.
    const    -> compile-time constant; value must be known at compile time.
    dynamic  -> opts OUT of static type checking (errors deferred to runtime).
    Object?  -> can hold anything, but type-safe (must check/cast to use members).
- Type inference also works with collections: `var list = [1,2,3]` -> List<int>.
- Everything in Dart is an OBJECT, including numbers, functions, and null.
  Every type ultimately extends `Object` (non-null) / `Object?` (nullable).

KEY DISTINCTIONS
----------------
var vs dynamic:
  - var  : type is inferred ONCE and locked. Compile-time safe.
  - dynamic : no type checking at all; you can call any method (fails at runtime
              if it doesn't exist).
  var x = 5;     x = 'hi';  // ERROR
  dynamic y = 5; y = 'hi';  // OK (no safety)

dynamic vs Object?:
  - Both can hold any value.
  - dynamic: compiler lets you call ANY member (unsafe).
  - Object?: compiler only lets you call Object members (toString, hashCode);
             you must cast/check before calling type-specific members. SAFER.

WHY Object? IS A SAFE ALTERNATIVE TO dynamic
---------------------------------------------
What `Object? x` means:
  - Object is the parent type of almost everything in Dart.
  - `?` means it can also be null.
  - So Object? can hold ANY value or null:
        Object? value = 'Hello';
        value = 123;
        value = true;
        value = null;        // all valid

The difference in safety:
  dynamic value = 123;
  print(value.length);   // COMPILES, then CRASHES at runtime (int has no length)

  Object? value = 123;
  print(value.length);   // COMPILE-TIME ERROR:
                         // The getter 'length' isn't defined for the type 'Object'

  Why? Dart only knows `value` is an Object, and not every object has `.length`.
  You must CHECK the type before using type-specific members:
        Object? value = 'Hello';
        if (value is String) {
          print(value.length);   // 5 — inside the if, Dart promotes value to String
        }

Analogy:
  - dynamic = a box labelled "trust me, something's inside" -> use it blindly,
    maybe it works, maybe it crashes.
  - Object? = a box labelled "some object is inside, but check what it is first"
    -> you inspect (`is`) before using it.

Why does this matter in a large codebase?
  dynamic data = getData();
  data.someMethod();        // compiler can't help
  data.someOtherMethod();   // any typo / missing member slips through
  data.length;              // ...all blow up at run time

  Object? data = getData();
  data.someMethod();        // Error — compiler forces you to check the type
  data.length;              // Error
  Mistakes are caught BEFORE running the app, not after it ships.

Bottom line: Object? forces you to verify the type before calling members that
may not exist, so mistakes are caught at COMPILE TIME instead of crashing later.

INTERVIEW Q&A
-------------
Q1. Difference between final and const?
A.  final = set once, value computed at RUNTIME (e.g. DateTime.now() is fine).
    const = COMPILE-TIME constant; value must be known when compiling, and the
    object is deeply immutable. const objects are CANONICALIZED — two identical
    const values are the SAME instance in memory (identical() returns true).

Q2. Can you have a const final? Difference between `const x` and `final x`?
A.  `const` implies compile-time + deeply immutable + canonicalized.
    `final` is just "assign once". A const variable is also implicitly final,
    but not vice versa. Use const when the value is a literal known at compile
    time; use final for runtime-computed values you won't reassign.

Q3. [GOTCHA] Is `const` a deep or shallow immutability?
A.  Deep. A const list/map cannot be modified at all, and its elements must
    themselves be const. `final` only prevents REASSIGNING the variable — the
    object it points to can still be mutated (e.g. final list.add(x) works).

Q4. What is the difference between var, dynamic and Object?
A.  var = inferred static type, fixed after inference (type-safe).
    dynamic = no compile-time checking, any member call allowed (unsafe).
    Object? = holds anything but only Object members callable without a cast.

Q5. What's the supertype of int and double?
A.  num. So `num x = 5; x = 5.5;` is valid. int and double both extend num.

Q6. [GOTCHA] What is `identical()` vs `==`?
A.  == checks VALUE equality (can be overridden). identical() checks REFERENCE
    equality (same object in memory). const canonicalization makes identical()
    true for equal const literals.

Q7. What is compile time vs run time? What's the difference?
A.  COMPILE TIME = when your source code is being compiled/analyzed (before the
    program runs). The compiler checks types, syntax, and rules here.
    RUN TIME = when the compiled program is actually executing on a device.
    Key differences:
      - WHEN: compile time happens first (build); run time happens after (launch).
      - ERRORS: compile-time errors stop the build (e.g. type mismatch, calling a
        member that doesn't exist on a known type) — you see them in the IDE
        before running. Run-time errors happen while the app runs (e.g. null
        check on null, dividing by zero, a failed `!` assertion) and can crash
        the app for the user.
      - SAFETY: compile-time errors are SAFER/cheaper — caught early. Run-time
        errors reach the user, so they're worse.
    This is the whole point of static typing & null safety: move as many errors
    as possible from RUN TIME to COMPILE TIME. (e.g. `const` is compile-time,
    `final` is run-time; `Object?` errors at compile time, `dynamic` at run time.)

===============================================================================
 SECTION 2 — NULL SAFETY (?, !, ??, ??=, late)
===============================================================================

NOTES
-----
- Dart has SOUND null safety: if a type is non-nullable, the value is GUARANTEED
  non-null at runtime. The compiler enforces it — no surprise null crashes.
- Non-nullable (default): `int x = 5;` — must be initialized, never null.
- Nullable: add `?` -> `int? x;` — can be a value OR null. Defaults to null.

OPERATORS
---------
  ?.   null-aware access  -> obj?.member ; whole expression is null if obj null.
  ??   if-null            -> a ?? b ; returns b when a is null.
  ??=  null-aware assign  -> a ??= b ; assigns b only if a is currently null.
  !    null assertion     -> a! ; tells compiler "trust me, not null".
                             THROWS at runtime if a is actually null.
  ?..  null-aware cascade
  ...? null-aware spread (skip if the collection is null)

late KEYWORD (two uses)
-----------------------
  1) Deferred init of a NON-NULLABLE: you promise to set it before first read.
     Reading it before assignment throws LateInitializationError.
       late String name;  name = 'x';  // ok
  2) LAZY initialization: the initializer runs ONLY on first access, then caches.
       late final config = expensiveSetup(); // runs on first use only

FLOW ANALYSIS / TYPE PROMOTION [GOTCHA]
---------------------------------------
- The compiler "promotes" a nullable to non-null after a null check:
      if (x != null) { x.length; }  // x is treated as non-null inside the if
- Promotion does NOT work on non-local / instance fields (another method could
  change them). Use a local variable or `!` for those:
      final v = obj.field; if (v != null) { v.use(); }

MEMORY TRICK
------------
  ?  = "maybe null"            (declare a type as nullable)
  ?. = "use it only if not null"  (else the whole expression is null)
  ?? = "or this default"       (fallback when the left side is null)
  !  = "definitely not null — trust me"  (THROWS if you're wrong)

DANGER OF `!`  [GOTCHA]
-----------------------
  String? name = null;
  print(name!.length);   // Runtime error: Null check operator used on a null value
Using `!` on a value that IS null throws immediately — you "lied" to Dart, so the
compile-time guarantee you bypassed turns into a runtime crash.

WHY DOES `!` EVEN EXIST?  (the real production reason) [GOTCHA]
--------------------------------------------------------------
If `?.` is safe and `!` can crash, why use `!` at all? Because `!` is NOT a tool
for handling null — it's a tool for ASSERTING AN INVARIANT the compiler can't
prove. It means: "I know something you don't: this cannot be null here."

There are exactly two situations, and they pick the operator for you:

  1) NULL IS EXPECTED / VALID  ->  use ?. , ?? , or `if (x != null)`
       user?.profileImage          // some users have no photo — that's normal
       token ?? 'guest'            // missing token has a sensible default
     Null is part of normal app flow; handle it gracefully.

  2) NULL WOULD MEAN A BUG  ->  use `!`
       FirebaseAuth.instance.currentUser!.uid   // we just signed in
       ModalRoute.of(context)!.settings         // this screen needs a route
     If it IS null, your app state is already broken. You WANT it to crash
     loudly and immediately (fail-fast) so you find the bug — instead of `?.`
     silently swallowing it and corrupting state further downstream.

So `!` turns a hidden, silent logic error into a visible, instant one. That is a
feature, not a risk — when the assumption is genuinely guaranteed elsewhere.

Two places `!` is legitimately needed:
  - Instance fields that won't promote: you checked it, but the compiler can't
    trust the field stayed non-null across a call ->
        if (user.name != null) { doStuff(); print(user.name!.length); }
    (cleaner: copy to a local — `final n = user.name; if (n != null) n.length;`)
  - Enforcing an invariant a layer above guarantees ->
        HomeScreen(user: currentUser!)   // login screen guarantees a user exists

Senior-dev rule of thumb:
  - "Can null happen during normal usage?"  -> YES -> ?. / ?? / null-check.
  - "Would null mean my app is broken?"     -> YES -> `!` (fail fast on purpose).
Never sprinkle `!` to silence the analyzer; put it only AFTER the value is truly
guaranteed (post-login, required constructor arg, validated input).

INTERVIEW Q&A
-------------
Q1. What does sound null safety mean?
A.  "Sound" = the type system's non-null guarantee can be trusted at runtime.
    The compiler statically proves a non-nullable value is never null, so there
    are no hidden null-dereference crashes from typed code.

Q2. Difference between `?` and `!`?
A.  `?` makes a type nullable (declares it CAN be null).
    `!` is a null assertion that strips nullability at a use site (asserts it
    is NOT null right now). `!` throws if you're wrong; `?` never throws.

Q3. When does `!` throw, and why is it risky?
A.  It throws a runtime error (Null check operator used on a null value) when
    the value is actually null. Risky because it bypasses the compiler's safety
    — only use it when YOU know more than the type system can prove.

Q4. Difference between `??` and `??=`?
A.  `a ?? b` is an expression returning a or b (doesn't change a).
    `a ??= b` is an assignment — sets a = b only if a was null.

Q5. What are the two uses of `late`?
A.  (1) Initialize a non-nullable variable later (deferred), promising to set it
    before reading. (2) Lazy initialization — defer an expensive initializer
    until first access, then cache the result.

Q6. [GOTCHA] late vs nullable (int? vs late int) — when to use which?
A.  Use `int?` when null is a VALID state (truly optional / may be absent).
    Use `late int` when the value is always present but set after construction
    (e.g. in initState), and you want non-null typing without `?` everywhere.
    late trades a compile-time guarantee for a runtime check.

Q7. What happens if you read a `late` variable before assigning it?
A.  Throws LateInitializationError at runtime.

Q8. Why might type promotion fail on a class field?
A.  Because a non-final field could be mutated between the null-check and use
    (even by another isolate/method), so the compiler can't guarantee it stays
    non-null. Copy to a local, or make the field `final`, or use `!`.

===============================================================================
 SECTION 3 — FUNCTIONS
===============================================================================

NOTES
-----
- Functions are FIRST-CLASS objects: assign to variables, pass as args, return.
- Parameter kinds:
    Positional required : fn(a, b)
    Optional positional : fn(a, [b, c])   -> in [], must be nullable or have default
    Named               : fn({a, b})      -> in {}, optional by default
    Named required      : fn({required a}) -> caller MUST pass it
- Default values: fn({String g = 'Hi'}) or fn(a, [int b = 0]).
- Arrow syntax: `int sq(int n) => n * n;` is sugar for `{ return n*n; }`
  (only for a SINGLE expression).
- A function with no explicit return returns null (declared return type `void`
  or the inferred type).
- Closures: an inner function captures variables from its enclosing scope and
  keeps them alive ("remembers" them).
- typedef: a named alias for a function signature, e.g.
      typedef IntTransform = int Function(int);
  Used heavily in Flutter (VoidCallback, ValueChanged<T>, etc.).

WORKED EXAMPLE — makeAdder (closure + typedef, see 02_functions.dart)
---------------------------------------------------------------------
Problem: writing addFive / addTen / addTwenty is the same code repeated.
Idea: write ONE function that BUILDS adder functions for us.

  typedef IntTransform = int Function(int value);  // "takes an int, returns an int"

  IntTransform makeAdder(int amount) {
    return (int value) => value + amount;   // returns a FUNCTION
  }
  // beginner-friendly equivalent of the body:
  //   int add(int value) => value + amount;
  //   return add;

Trace it:
  final addTen = makeAdder(10);   // amount = 10, returns (value) => value + 10
  addTen(5);                      // value = 5  ->  5 + 10  =  15

  final addTwenty = makeAdder(20);// amount = 20, returns (value) => value + 20
  addTwenty(5);                   // 5 + 20  =  25

Why does addTen still know amount = 10 after makeAdder finished?
  Normally a function's locals vanish when it returns. But the returned function
  STILL USES `amount`, so Dart keeps it alive. That captured-and-remembered
  variable is a CLOSURE. Each adder remembers its OWN amount:
        addTen    -> remembers amount = 10
        addTwenty -> remembers amount = 20

Analogy: makeAdder(10) is a machine pre-configured to "+10"; feed it 5, get 15.
makeAdder(20) is a different machine ("+20"); each remembers how it was built.

WORKED EXAMPLE — makeCounter (a closure over MUTABLE state, see 02_functions.dart #9)
-------------------------------------------------------------------------------------
makeAdder remembered a FIXED value; makeCounter remembers a variable it keeps
CHANGING. This is the clearest proof that the captured variable lives on.

  int Function() makeCounter() {
    int count = 0;                 // local to makeCounter
    return () => ++count;          // returned closure captures `count`
  }

  final counter = makeCounter();
  print(counter());   // 1
  print(counter());   // 2
  print(counter());   // 3

Trace: each call runs `++count` (increment THEN return) on the SAME count:
  call 1: count 0 -> 1, returns 1
  call 2: count 1 -> 2, returns 2
  call 3: count 2 -> 3, returns 3

"Why doesn't count reset to 0 each call?"
  Normally a function's locals vanish when it returns, so you'd expect count to
  be gone. But the returned closure STILL USES count, so Dart keeps that one
  variable alive across calls — it is NOT recreated. The closure reads and
  writes the same `count` every time, so it accumulates: 1, 2, 3, ...
  Analogy: makeCounter() hands you a notebook starting at 0; each counter() call
  reads it, adds 1, writes it back, and returns the new number.

[GOTCHA] Each call to makeCounter() makes a NEW, independent count:
  final a = makeCounter(); final b = makeCounter();
  a(); a();   // 1, 2
  b();        // 1  — b has its OWN count, unaffected by a.

PASSING A FUNCTION AS AN ARGUMENT — how the lambda "connects" [common confusion]
--------------------------------------------------------------------------------
The thing that confuses everyone about groupBy(people, (p) => p.city):
HOW does `(p) => p.city` connect to the parameter `String Function(Person) getKey`
and to the later call `getKey(person)`? Forget groupBy and build it up:

  1) A function CALL just substitutes the argument:
        void greet(String name) => print('Hello $name');
        greet('Aditya');     // name = 'Aditya' -> Hello Aditya

  2) A function is a VALUE you can store (no () = the function itself):
        String getCity(Person p) => p.city;
        var fn = getCity;            // fn now POINTS to getCity
        fn(alice);                   // identical to getCity(alice)

  3) So you can PASS a function into another function. The parameter's type just
     describes the shape of the function it accepts:
        void execute(String Function(String) f) => print(f('Aditya'));
        String shout(String s) => s.toUpperCase();
        execute(shout);              // f = shout; f('Aditya') -> 'ADITYA'
     `String Function(String)` reads as "a function taking a String, returning a
     String". (Pass the reference `shout`, NOT `shout()` — see the () gotcha below.)

  4) Now groupBy's 2nd parameter `String Function(Person) getKey` is the SAME idea
     — "a function taking a Person, returning a String." When you call:
        groupBy(people, getCity);          // getKey = getCity
     then inside groupBy `final key = getKey(person);` is literally `getCity(person)`.

  5) The lambda is just an unnamed version of that function. These two are equal:
        groupBy(people, getCity);
        groupBy(people, (p) => p.city);    // getKey = (p) => p.city
     Tracing one iteration with the lambda:
        item = alice
        key  = getKey(item)
             = ((p) => p.city)(alice)      // substitute p = alice
             = alice.city
             = 'Mumbai'

MENTAL MODEL — the callback is a "remote control": groupBy doesn't know whether to
group by city, age, or department. It asks the function YOU handed it, once per
item: "which bucket does this go in?" Whatever that function returns becomes the
map key. This is the essence of a higher-order function — and the same mechanism
behind map, where, sort's compare, fold, forEach, and every Flutter callback.

FIRST-CLASS FUNCTION vs CLOSURE (two different concepts, often confused)
-----------------------------------------------------------------------
makeAdder shows BOTH at once — but they are NOT the same thing:

FIRST-CLASS FUNCTION = the language treats functions like ordinary values.
  In Dart a function can be:
    - stored in a variable :  var op = add;        op(2, 3);
    - passed as an argument:  list.forEach(print); execute(sayHello);
    - returned from a function: makeAdder(...) returns a function
  Because functions can be used anywhere a value can, they are "first-class
  citizens". This is a property of the LANGUAGE.

CLOSURE = a returned/inner function that REMEMBERS variables from the scope
  where it was created, even after that outer scope has finished.
    final addTen = makeAdder(10);  // addTen keeps `amount = 10` alive
  This is a property of a specific FUNCTION VALUE (its captured state).

How they combine in makeAdder:
    final addTen = makeAdder(10);
  - FIRST-CLASS: makeAdder can return a function and we store it in addTen.
  - CLOSURE    : that returned function still remembers amount = 10.
  First-class makes "return/store a function" legal; closure is WHY the stored
  function still knows `amount`. You can have first-class functions without a
  closure (e.g. `var op = add;` captures nothing).

One-line interview answers:
  First-class function -> "Functions can be stored in variables, passed as
                           arguments, and returned from other functions."
  Closure              -> "A function that remembers and can access variables
                           from its outer scope even after that scope has
                           finished executing."

CALLING WITH `()` vs WITHOUT `()`  (reference vs invocation) [GOTCHA]
--------------------------------------------------------------------
  fn      -> the FUNCTION ITSELF (a value you can store/pass). Does NOT run.
  fn()    -> CALLS/EXECUTES the function (and you get its return value).

  void sayHello() => print('Hello');
  var x = sayHello;    // x now HOLDS the function (nothing printed yet)
  x();                 // NOW it runs -> Hello
  sayHello();          // runs -> Hello

When you PASS a function, pass the reference (no ()): the receiver calls it later.
  void execute(Function fn) => fn();
  execute(sayHello);   // correct: hand over the function, execute() runs it
  execute(sayHello()); // WRONG: runs sayHello() FIRST, then passes its result
                       // (here null) into execute -> bug / type error.

Quick table:
  sayHello             function itself (reference)
  sayHello()           execute it
  var x = sayHello     store the function
  var x = sayHello()   store the RETURNED VALUE
  onPressed: sayHello  pass the function (Flutter calls it on tap)  ✅
  onPressed: sayHello()runs immediately while BUILDING the UI       ❌

FLUTTER [GOTCHA]: callbacks like onPressed / onTap / onChanged / builder /
itemBuilder expect a function REFERENCE (myFunction), NOT a call (myFunction()).
Writing `onPressed: myFunction()` runs it during build instead of on the event —
one of the most common beginner mistakes.
  Ask: "Run it NOW?" -> use ()   |   "Pass/store for LATER?" -> no ()

[GOTCHA] You CANNOT mix optional positional [ ] and named { } in the same fn.

INTERVIEW Q&A
-------------
Q1. Difference between optional positional and named parameters?
A.  Optional positional [ ]: order matters, omitted from the right; good for a
    small number of natural args. Named { }: order-independent, self-documenting
    at the call site; can be marked `required`. Named is preferred for clarity,
    especially with 2+ optional args (this is why Flutter widgets use named).

Q2. Are named parameters required or optional by default?
A.  Optional by default. Add `required` to force the caller to supply them, or
    give a default value.

Q3. What is a closure? Give an example.
A.  A function that captures and retains variables from its lexical scope.
    Example: a counter — `int Function() makeCounter() { int c=0; return ()=>++c; }`
    Each returned closure keeps its own `c` alive across calls.

Q4. What is a higher-order function?
A.  A function that takes another function as a parameter and/or returns a
    function. e.g. List.map, List.where, or makeAdder(int) returning a closure.

Q5. What is the difference between `=>` and `{}` function bodies?
A.  `=>` is for a single expression and implicitly returns its value.
    `{}` is a block body that can contain multiple statements; you must write
    `return` explicitly to return a value.

Q6. What is a typedef and why use it?
A.  An alias for a type (commonly a function signature). It improves readability
    and reuse for callback types. e.g. `typedef Validator = bool Function(String);`

Q7. [GOTCHA] What does a function with return type `void` return?
A.  Conceptually nothing; you should not use its result. (Internally null, but
    you shouldn't rely on that.)

===============================================================================
 SECTION 4 — COLLECTIONS (List, Set, Map) + Iterable methods
===============================================================================

NOTES
-----
LIST  : ordered, indexable, allows duplicates. Default growable.
SET   : unordered (insertion-order in practice), UNIQUE elements, O(1) lookup.
MAP   : key/value pairs, unique keys, value lookup by key.

Common List ops: add, insert, remove (by value), removeAt (by index), indexOf,
  contains, first, last, length, sort, sublist, reversed.
Set ops: union, intersection, difference, contains, add (returns bool).
Map ops: [], []=, putIfAbsent, update, remove, containsKey, keys, values, entries.

ITERABLE METHODS (know these COLD):
  map(f)      -> transform each element; NEW iterable, SAME length.
  where(p)    -> filter; keeps elements where predicate is true.
  fold(seed,f)-> reduce to ONE value with an initial seed; result type can DIFFER.
  reduce(f)   -> like fold but seed = first element; THROWS on empty list.
  forEach(f)  -> side effects only; returns nothing.
  any(p)/every(p) -> boolean checks.
  expand(f)   -> flatMap: map each element to an iterable, then flatten.
  firstWhere/lastWhere/singleWhere -> find with predicate (orElse to avoid throw).
  take(n)/skip(n), toList(), toSet(), join(sep).

[GOTCHA] LAZINESS: map() and where() are LAZY — they return an Iterable and
  don't run until iterated (e.g. via toList() or a for-in). This matters for
  side effects and performance.

COLLECTION-IF / COLLECTION-FOR / SPREAD (very common in Flutter widget trees):
  [a, ...other, if (cond) b, for (var x in xs) x*2]

reduce() vs fold() — DEEP DIVE
------------------------------
Both COMBINE all elements of a collection into a SINGLE value. The difference is
WHERE the accumulation starts.

reduce(f)
  final product = nums.reduce((acc, n) => acc + n);
  - Uses the FIRST element of the list as the initial accumulator (acc).
  - Starts iterating from the SECOND element.
  - CANNOT be used on an empty list (throws a Bad state / No element error).

  List<int> nums = [10, 20, 30];
  final sum = nums.reduce((acc, n) => acc + n);
  // Step 1: acc = 10, n = 20 -> 30
  // Step 2: acc = 30, n = 30 -> 60
  print(sum); // 60

fold(seed, f)
  final sum = nums.fold(0, (acc, n) => acc + n);
  - YOU provide the initial value (the seed, 0 here).
  - Iteration starts from the FIRST element.
  - Works even on an EMPTY list (returns the seed).

  List<int> nums = [10, 20, 30];
  final sum = nums.fold(0, (acc, n) => acc + n);
  // Step 1: acc = 0,  n = 10 -> 10
  // Step 2: acc = 10, n = 20 -> 30
  // Step 3: acc = 30, n = 30 -> 60
  print(sum); // 60

EMPTY-LIST BEHAVIOR [GOTCHA]
  List<int> nums = [];
  nums.reduce((acc, n) => acc + n);        // ERROR (no first element to seed)
  final sum = nums.fold(0, (acc, n) => acc + n);
  print(sum);                              // 0  (just the seed)

fold() CAN RETURN A DIFFERENT TYPE (reduce CANNOT)
  reduce's accumulator IS an element, so the result type must match the element
  type. fold's accumulator is the seed, so it can be ANY type.

  List<int> nums = [1, 2, 3];
  String result = nums.fold('Numbers:', (acc, n) => '$acc $n');
  print(result);   // Numbers: 1 2 3
  // element type = int, result type = String

WHEN TO USE WHICH
  Combine elements into the SAME type ........... reduce()
  Need an explicit initial value ................ fold()
  List MIGHT be empty ........................... fold()
  Result type DIFFERS from element type ......... fold()

QUICK EXAMPLES
  nums.fold(0, (acc, n) => acc + n);        // sum
  nums.fold(1, (acc, n) => acc * n);        // product
  nums.reduce((a, b) => a > b ? a : b);     // maximum
  nums.fold('', (acc, n) => '$acc$n ');     // build a String

RULE OF THUMB: reduce() = fold() without an explicit initial value. If unsure,
fold() is the safer, more flexible choice (empty-safe + any result type).

FINDING MAXIMUM — reduce() vs fold() SIDE BY SIDE
-------------------------------------------------
With reduce() (first element is a natural starting point):
  List<int> nums = [5, 2, 8, 3, 6];
  final max = nums.reduce((a, b) => a > b ? a : b);
  print(max); // 8

With fold() seeding from the first element:
  List<int> nums = [5, 2, 8, 3, 6];
  final max = nums.fold(nums.first, (a, b) => a > b ? a : b);
  print(max); // 8

  Trace (acc starts at 5):
    5 vs 5 -> 5
    5 vs 2 -> 5
    5 vs 8 -> 8
    8 vs 3 -> 8
    8 vs 6 -> 8
  Result: 8

A better fold() seed — if you know a floor value, seed with it instead:
  final max = nums.fold(-999999, (a, b) => a > b ? a : b);

  // cleanest with dart:math:
  import 'dart:math';
  final max = nums.fold(-double.infinity, (a, b) => max(a, b));

So why use reduce() here at all?
  When you combine elements of the SAME type and the first element is a natural
  starting value, reduce() is cleaner — no seed to invent:
    nums.reduce((a, b) => a > b ? a : b);          // clean
    nums.fold(nums.first, (a, b) => a > b ? a : b); // same result, more noise

RULE OF THUMB (max/min flavour):
  - First element can act as the start -> reduce().
  - Need a custom start value or a different result type -> fold().

  // Sum
  nums.reduce((a, b) => a + b);
  nums.fold(0, (a, b) => a + b);
  // Max
  nums.reduce((a, b) => a > b ? a : b);
  nums.fold(nums.first, (a, b) => a > b ? a : b);
  // List -> String  (IMPOSSIBLE with reduce: result type String != element int)
  nums.fold('', (a, b) => '$a$b');

GROUP BY A KEY — putIfAbsent DEEP DIVE (see 04_problems.dart #3)
---------------------------------------------------------------
Goal: turn a flat list into a Map<Key, List<Value>> (a "bucket per key"). This
is THE classic interview pattern and the engine behind package:collection groupBy.

  Map<K, List<T>> groupBy<T, K>(Iterable<T> items, K Function(T) keyOf) {
    final result = <K, List<T>>{};
    for (final item in items) {
      final key = keyOf(item);
      result.putIfAbsent(key, () => <T>[]).add(item);   // <- does all the work
    }
    return result;
  }

  final byCity = groupBy(people, (p) => p.city);  // K=String, T=Person
  // {Mumbai: [Alice, Carol], Delhi: [Bob, Dan], Pune: [Eve]}

The one line that matters:
  result.putIfAbsent(key, () => <T>[]).add(item);
  = "get the bucket list for this key (creating an empty one if it's the first
     time we've seen the key), then append item to it."
  Verbose equivalent:
    if (!result.containsKey(key)) result[key] = <T>[];
    result[key]!.add(item);

Two subtle points interviewers probe [GOTCHA]:
  1) The 2nd arg is a FUNCTION `() => <T>[]`, not a value `<T>[]`. It's LAZY —
     the empty list is built ONLY when the key is missing. Passing a plain
     `<T>[]` would allocate a throwaway list on every iteration.
  2) putIfAbsent RETURNS the value either way: the new list on a fresh key, the
     existing list on a repeat key. That's why one line handles both the
     "Mumbai first time" and "Mumbai again" cases with no if-statement.
  Note: `result[key]!` needs `!` because a map lookup is typed `List<T>?`
  (lookups can be null) — the containsKey check above guarantees it's non-null.

  BEGINNER MENTAL MODEL (ignore the generics <T,K> at first):
    Think of it like sorting students into classrooms by city. Walk the list one
    person at a time; the city is the classroom name. If that classroom doesn't
    exist yet, open an empty one; then put the person inside it.
      Map<String, List<Person>> result = {};
      for (var person in people) {
        String city = person.city;
        if (!result.containsKey(city)) result[city] = [];   // open classroom
        result[city]!.add(person);                          // seat the person
      }
    The generic version is the SAME four steps — take a person, get the key, make
    the bucket if missing, add to it — just written to work for ANY type, not only
    Person/String. Get comfy with this loop first; then putIfAbsent reads as the
    short form of the if-check + add.

  READING THE SIGNATURE — what <T, K> and `K Function(T)` mean:
    Map<K, List<T>> groupBy<T, K>(Iterable<T> items, K Function(T) keyOf)
    - <T, K> are TYPE PLACEHOLDERS filled in per call (like function args, but for
      types). T = type of each item; K = type of the grouping key.
    - Dart INFERS them from the arguments — you rarely write them at the call site:
        groupBy(people,  (p) => p.city)    // T=Person, K=String -> Map<String, List<Person>>
        groupBy(numbers, (n) => n.isEven)  // T=int,    K=bool   -> Map<bool,   List<int>>
        // {true: [2,4], false: [1,3,5]}
        groupBy(fruits,  (f) => f[0])      // T=String, K=String -> Map<String, List<String>>
        // {A: [Apple, Avocado], B: [Banana, Berry]}
    - `K Function(T) keyOf` is a FUNCTION-TYPE parameter: "a function that takes a T
      and returns a K." For people that's `String Function(Person)`. The lambda
      `(p) => p.city` IS such a function — shorthand for:
          String getCity(Person p) => p.city;
      so groupBy(people, (p) => p.city) == groupBy(people, getCity).
    - keyOf answers ONE question per item: "which bucket does this go into?"
    This is why ONE generic function replaces a separate group-people-by-city,
    group-numbers-by-even, group-fruits-by-letter, etc. (links to Section 3's
    first-class functions + typedef.)

  [GOTCHA] You MUST declare T/K with <...> before you use them. If a type name
  like T or K appears anywhere in a signature, it has to be introduced by a
  <...> clause first — otherwise the compiler doesn't know what it means.
    correct:  Map<K, List<T>> groupBy<T, K>(Iterable<T> items, K Function(T) keyOf)
    wrong:    Map<K, List<T>> groupBy(Iterable<T> items, K Function(T) keyOf)
              // ERROR: 'T'/'K' undefined — never declared.
    Same idea as ordinary variables: `print(x)` fails if x was never declared;
    `Map<K, List<T>>` fails if <T, K> was never declared.
    Where the <...> goes:
      - function/method:  T identity<T>(T value) => value;   // identity<int>(10)
      - class:            class Box<T> { T value; ... }
      - bound (optional): <T extends num>  // T must be a num subtype
    Escape hatch: if you DON'T need it reusable, drop generics and use concrete
    types — then no <...> is needed because nothing is a placeholder:
        Map<String, List<Person>> groupBy(
            List<Person> people, String Function(Person) getKey) { ... }
    Trade-off: that version works ONLY for Person/String; the generic one works
    for any item+key type. Generics buy reuse at the cost of the <...> ceremony.

  [GOTCHA] The buckets hold the FULL items, not the key-derived display value:
    final byCity = groupBy(people, (p) => p.city);
    // byCity['Mumbai'] is [Person(Alice), Person(Carol)] — whole Person objects,
    // NOT ['Alice','Carol']. Type is Map<String, List<Person>>.
    You only see names in the printout because of the .map at print time:
        print('$city: ${group.map((p) => p.name).toList()}');  // List<Person> -> List<String>
    That transform is just for DISPLAY; the map still stores complete Person objects.
    - print('$city: $group') without the .map shows: [Instance of 'Person', ...]
      because Person has no toString(). Add one to see real details:
          @override
          String toString() => 'Person(name: $name, city: $city)';
    (Same idea as Section 1: a class prints as "Instance of 'X'" until you override
    toString().)

  HOW THIS FUNCTION EVOLVES (5 stages — how a dev grows into the generic version):
    Stage 1 — Hardcoded logic, inline in main(). Just solve it:
        Map<String, List<Person>> result = {};
        for (var p in people) {
          if (!result.containsKey(p.city)) result[p.city] = [];
          result[p.city]!.add(p);
        }
    Stage 2 — Extract into a function (input -> output, reusable call site):
        Map<String, List<Person>> groupPeopleByCity(List<Person> people) { ...same... }
    Stage 3 — Replace the if-check with putIfAbsent (cleaner):
        result.putIfAbsent(p.city, () => []);
        result[p.city]!.add(p);
    Stage 4 — Pass the KEY rule as a parameter, so it's not locked to city:
        Map<String, List<Person>> groupBy(
            List<Person> people, String Function(Person) getKey) { ... }
        groupBy(people, (p) => p.city);   // today city, tomorrow p.age, etc.
      (still Person-only; the only new idea is "behavior as a parameter".)
    Stage 5 — Make item AND key types generic, so it works for ANY collection:
        Map<K, List<T>> groupBy<T, K>(Iterable<T> items, K Function(T) keyOf) { ... }
        groupBy(people, (p) => p.city)     // Map<String, List<Person>>
        groupBy(nums,   (n) => n.isEven)   // Map<bool,   List<int>>
        groupBy(words,  (w) => w[0])       // Map<String, List<String>>
    The arc: hardcoded -> function -> cleaner -> behavior-as-parameter -> generic.
    Each stage changes ONE thing; the loop's core idea never changes.

REVERSE A STRING — split('').reversed.join() DEEP DIVE (see 04_problems.dart #1)
-------------------------------------------------------------------------------
Goal: reverse a string. Dart strings are immutable and not directly indexable as
a list, so the idiomatic move is: chars -> reverse -> rejoin.

  void reversedString(String val) {
    print(val.split('').reversed.join());
  }
  reversedString("hello");   // olleh

Step by step (with the TYPE at each stage — interviewers love this):
  Expression                       Result                   Type
  "hello"                          "hello"                  String
  .split('')                       [h, e, l, l, o]          List<String>
  .reversed                        (o, l, l, e, h)          Iterable<String>  <- NOT a List!
  .join()                          "olleh"                  String

  - split('')  : breaks the string into a list of single-character strings.
  - reversed   : returns a LAZY Iterable in reverse order (no new list built yet).
  - join()     : concatenates all elements into one String. join('-') inserts a
                 separator -> "o-l-l-e-h".

Verbose equivalent (same thing, named locals):
  void reversedString(String val) {
    List<String> chars        = val.split('');
    Iterable<String> reversed = chars.reversed;
    String result             = reversed.join();
    print(result);
  }

[GOTCHA] split('') is NOT Unicode-correct. It splits by UTF-16 code units, so
emoji and combined characters (e.g. flags, skin-tone emoji, accented letters made
of base+combining marks) get torn apart and reversed wrongly.
  - ASCII / simple text  -> split('').reversed.join()  is the expected answer.
  - Unicode-safe         -> use runes, or `characters` from package:characters:
        import 'package:characters/characters.dart';
        final r = text.characters.toList().reversed.join();  // grapheme-safe
Mention this caveat in an interview — it shows you know strings aren't just bytes.

INTERVIEW Q&A
-------------
Q1. Difference between List, Set and Map?
A.  List = ordered, indexed, duplicates allowed. Set = unique, no index, fast
    membership test. Map = key->value, unique keys. Choose Set for dedup/contains,
    Map for lookups, List for ordered sequences.

Q2. Difference between map(), where(), and fold()?
A.  map transforms each element (same count); where filters; fold collapses the
    whole iterable into a single accumulated value using a seed.

Q3. Difference between fold() and reduce()?
A.  reduce uses the FIRST element as the initial value and THROWS on an empty
    collection; the result type matches the element type. fold takes an explicit
    seed (safe on empty) and the result type can be different from the element.

Q4. [GOTCHA] Are map/where eager or lazy?
A.  Lazy. They return an Iterable computed on demand. Materialize with toList()/
    toSet(), or iterate, to actually run the transformations.

Q5. How do you remove duplicates from a List?
A.  list.toSet().toList(). (Order is generally preserved for LinkedHashSet, the
    default.)

Q6. How do you safely read a missing map key?
A.  map[key] returns null if absent; use `map[key] ?? defaultValue`, or
    putIfAbsent / containsKey. Note the lookup return type is the value type
    made nullable.

Q7. Difference between putIfAbsent and update on a Map?
A.  putIfAbsent(key, () => v) inserts only when the key is missing.
    update(key, (old)=>new, ifAbsent: ()=>v) modifies an existing value, with
    an optional fallback to create one if absent.

Q8. What is the difference between expand() and map()?
A.  map: 1 element -> 1 element. expand (flatMap): 1 element -> an iterable,
    then all results are flattened into one iterable.

NOTES
-----
- CLASS members: fields, constructors, getters/setters, methods, static members.
- PRIVACY: a leading underscore `_name` makes a member private to its LIBRARY
  (file), NOT to the class. Dart has no public/private/protected keywords.
- CONSTRUCTORS:
    Default          : ClassName(this.a, this.b);  // shorthand assigns fields
    Named            : ClassName.fromJson(...) {}
    Initializer list : ClassName(...) : _x = v, assert(...) { body }
                       runs BEFORE the body; used for final fields, asserts,
                       and calling super(...).
    const constructor: const ClassName(...) -> enables compile-time instances.
    Factory          : factory ClassName(...) -> can return cached/subtype
                       instances; does NOT always create a new object.
- GETTERS/SETTERS:
    int get area => w * h;   // computed, read-only if no setter
    set name(String v) => _name = v;
    Prefer a getter over a public field when the value is derived or you want
    read-only access (encapsulation).
- INHERITANCE: `class B extends A` ; single inheritance only.
    super(...) calls the parent constructor.
    @override to redefine a method; call super.method() to extend behavior.
- ABSTRACT classes: `abstract class Shape { double area(); }` — can't be
  instantiated; defines a contract + optional shared code.
- INTERFACES: every class is an implicit interface. `class A implements B`
  forces A to provide ALL of B's members (no implementation inherited).
- MIXINS: `mixin M {}` + `class C with M` — reuse code across unrelated classes
  WITHOUT inheritance. Resolves the diamond problem via linearization.
- POLYMORPHISM: a variable of the base type can hold any subtype; the actual
  method that runs is the subtype's override (dynamic dispatch).

[GOTCHA] extends vs implements vs with:
  extends    -> inherit implementation (one parent), can call super.
  implements -> inherit only the CONTRACT (must reimplement everything).
  with       -> mix in reusable implementation from a mixin.

INITIALIZER LIST ORDER [GOTCHA]:
  1) initializer list (field inits, asserts) runs first,
  2) then super constructor,
  3) then the constructor body.
  You cannot use `this` in the initializer list (object not built yet).

WORKED EXAMPLE — BankAccount -> SavingsAccount (see 05_oop_bank_account.dart)
----------------------------------------------------------------------------
ONE example that hits ~70-80% of OOP interview topics: class/object, fields,
encapsulation, constructor, initializer list, assert, getter, methods,
exceptions, inheritance, super, override, polymorphism, toString.

  CLASS = blueprint ; OBJECT = a thing built from it.
    final acc = BankAccount(owner: 'Aditya', openingBalance: 1000);  // acc is the object

  FIELDS + ENCAPSULATION:
    final String owner;   // public, set once
    double _balance;      // _ = library-private -> outside code CANNOT do acc._balance = 9
    Balance changes ONLY through deposit()/withdraw(), so it can't go inconsistent.

  CONSTRUCTOR + INITIALIZER LIST + ASSERT:
    BankAccount({required this.owner, double openingBalance = 0})
        : assert(openingBalance >= 0, 'Opening balance cannot be negative'),
          _balance = openingBalance;
    - `required this.owner` assigns the field straight from the named arg.
    - the part after `:` is the INITIALIZER LIST — runs BEFORE the body. assert
      blocks a negative opening balance; `_balance = openingBalance` seeds balance.
      (final fields like this MUST be set here or inline, not in the body.)

  GETTER (read-only public window onto a private field):
    double get balance => _balance;
    acc.balance;        // reads fine -> 1000
    acc.balance = 5000; // COMPILE ERROR — no setter, so it's read-only

  METHODS mutate through guarded entry points:
    deposit(500)  -> _balance += amount   (throws ArgumentError if amount <= 0)
    withdraw(200) -> _balance -= amount   (throws if amount<=0 or amount>balance)

  EXCEPTIONS + try/catch (fail loudly instead of corrupting state):
    withdraw(99999) -> throw InsufficientFundsException(...)
    try { acc.withdraw(99999); }
    on InsufficientFundsException catch (e) { print('Caught: $e'); }  // no crash
    (Custom type `implements Exception` -> a meaningful domain error, not a generic one.)

  INHERITANCE (`extends`) — SavingsAccount IS-A BankAccount:
    class SavingsAccount extends BankAccount { ... }
    It gets owner, balance, deposit(), withdraw() FOR FREE — no rewrite.
    Constructor forwards to the parent with `super.`:
        SavingsAccount({required super.owner, super.openingBalance,
                        required this.annualInterestRate});
    New behavior of its own:
        void applyMonthlyInterest() {            // balance 1500 @6%/yr
          deposit(balance * annualInterestRate / 12);  // +7.50 -> reuses parent deposit
        }

  OVERRIDE + super (tighten a rule, then defer to the parent):
    @override
    void withdraw(double amount) {
      if (balance - amount < minimumBalance)         // savings-only rule (keep >= 100)
        throw InsufficientFundsException(...);
      super.withdraw(amount);                        // then run base validation + debit
    }
    Base rule: withdraw if money exists. Savings rule: must keep >= 100. super.withdraw
    avoids duplicating the base checks. (static const minimumBalance = 100 -> shared by
    the class, not per-instance.)

  POLYMORPHISM — a base-typed list holding subclass instances:
    final List<BankAccount> portfolio = [acc, savings];  // SavingsAccount IS-A BankAccount
    final total = portfolio.fold<double>(0, (sum, a) => sum + a.balance);  // 1300 + 1507.50
    Treat both uniformly through the BankAccount type; the right toString()/withdraw()
    still runs per actual subtype (dynamic dispatch). (Like List<Animal> holding Dog & Cat.)

  toString() — without it `print(acc)` shows "Instance of 'BankAccount'"; overriding
  gives "BankAccount(Aditya): balance=1300.00". (Same Instance-of point as Section 1.)

  MAP OF CONCEPTS -> WHERE THEY LIVE HERE:
    Class/Object -> BankAccount / acc        Encapsulation -> private _balance
    Getter -> balance                        Inheritance   -> SavingsAccount extends BankAccount
    Override -> SavingsAccount.withdraw       Polymorphism  -> List<BankAccount> holds SavingsAccount
    Exception -> InsufficientFundsException   super -> forwards ctor + reuses withdraw

assert() — DEV-TIME SANITY CHECK (used in the BankAccount constructor above)
----------------------------------------------------------------------------
assert(condition[, message]) checks an assumption WHILE DEVELOPING:
  - condition true  -> nothing happens, program continues.
  - condition false -> throws AssertionError (with your message, if given) and halts,
    so you find the bug immediately.
    assert(age >= 18);                         // 15 -> AssertionError
    assert(age >= 18, 'Age must be at least 18'); // -> AssertionError: Age must be at least 18

In the constructor:
  BankAccount({required this.owner, double openingBalance = 0})
      : assert(openingBalance >= 0, 'Opening balance cannot be negative'), _balance = ...
  openingBalance: 1000 -> 1000 >= 0 true -> account created.
  openingBalance: -1000 -> false -> AssertionError, object never built.

[GOTCHA] assert is DEBUG-ONLY — it is STRIPPED from production (release / AOT)
builds and does nothing there. So NEVER use it to validate real user input or
guard security/business rules — those checks vanish in the shipped app.
  - assert  -> "this can only be false if I (the programmer) made a mistake."
               Catches MY bugs during development. Dev-only.
  - if + throw -> "this CAN happen at runtime from real input." Runs in production.
        if (amount <= 0) throw ArgumentError('must be positive');  // user-facing, always runs
  That's exactly why BankAccount uses assert for the programmer-invariant
  (openingBalance >= 0 at construction) but `if (...) throw` inside deposit/withdraw
  for real runtime amounts. (Ties to Section 1: assert = compile/dev-time guard,
  thrown exceptions = run-time and reach the user.)
  Common spots: constructors, Flutter widget invariants, validating internal
  assumptions. Rule: "must ALWAYS be true or it's a bug" -> assert; "might be false
  from real input" -> if + throw.

INTERVIEW Q&A
-------------
Q1. How is privacy implemented in Dart?
A.  By a leading underscore (_member). It's LIBRARY-private (per file), not
    class-private. Same-file code can access another class's _members.

Q2. Difference between a named constructor and a factory constructor?
A.  Named constructor: an additional constructor with a name; always creates a
    new instance via the normal mechanism. Factory: may return an existing
    instance (caching/singleton), a subtype, or compute which object to return —
    it doesn't have to create a fresh one and has no initializer list.

Q3. What is an initializer list and when do you need it?
A.  The `: field = value, assert(...)` part before the constructor body. Needed
    to initialize FINAL fields, run asserts, and call super(...) — all before
    the body runs and before `this` is available.

Q4. Difference between extends, implements, and with?
A.  extends = single inheritance of implementation (super available).
    implements = adopt the interface/contract only; must implement every member.
    with = mix in shared code from a mixin without an is-a relationship.

Q5. What is the difference between an abstract class and an interface in Dart?
A.  Dart has no separate `interface` keyword — every class defines an implicit
    interface. An abstract class can't be instantiated and may contain concrete
    methods + abstract ones; you `extend` it to inherit code or `implement` it
    to use only its contract.

Q6. What is a mixin and why use it over inheritance?
A.  A mixin shares method/field implementations across classes that aren't in
    the same hierarchy (composition over inheritance). Use it to reuse behavior
    without forcing an is-a relationship or fighting single inheritance.

Q7. When do you use a getter instead of a public field?
A.  When the value is computed/derived (e.g. area = w*h), or when you want
    read-only access while keeping the backing field private (encapsulation),
    or to add logic/validation without changing the call site.

Q8. What is polymorphism? Give a Dart example.
A.  The ability to treat subtypes through a base-type reference and have the
    correct overridden method run at runtime. e.g. List<BankAccount> holding a
    SavingsAccount; calling withdraw() runs SavingsAccount's override.

Q9. [GOTCHA] Why can't you use `this` in an initializer list?
A.  Because the object isn't fully constructed yet — fields are still being
    initialized and the super constructor hasn't run, so `this` is not valid.

Q10. How do you create an immutable class in Dart?
A.  Make all fields `final` and provide a `const` constructor. Then instances
    can be const and are deeply immutable (this is exactly how Flutter widgets
    like `const Text(...)` work, enabling widget reuse/perf).

===============================================================================
 RAPID-FIRE / CROSS-CUTTING QUESTIONS
===============================================================================

Q. Is Dart compiled or interpreted?
A. Both. JIT (Just-In-Time) compilation in development enables hot reload; AOT
   (Ahead-Of-Time) compiles to native machine code for fast release builds.
   Dart can also compile to JavaScript for the web.

Q. What is the entry point of a Dart program?
A. The top-level `main()` function (optionally `main(List<String> args)`).

Q. Does Dart support multiple inheritance?
A. Not via classes (single inheritance). Achieve similar reuse with mixins
   (`with`) and multiple `implements`.

Q. What is the `==` default behavior for objects?
A. By default == is identity (same reference). Override == and hashCode together
   for value equality. (Or use records / packages / freezed in real projects.)

Q. What is cascade notation (..)?
A. Lets you chain operations on the SAME object without repeating it:
     final p = Paint()..color = red ..strokeWidth = 2;
   Returns the object itself, not the method results.

Q. What is the difference between an Iterable and a List?
A. Iterable is the abstract sequence you can iterate once/lazily; List is a
   concrete, indexable, materialized Iterable with a length and random access.

Q. What is `dynamic` typing's main risk?
A. Errors move from compile time to RUNTIME — you lose autocomplete and the
   compiler's safety net. Prefer specific types or Object? + casts.

Q. What are records (Dart 3)?  [bonus, may impress]
A. Lightweight anonymous immutable aggregates: `(int, String) r = (1, 'a');`
   Great for returning multiple values without a class. Access via r.$1, r.$2
   or named fields.

Q. What is pattern matching / switch expressions (Dart 3)? [bonus]
A. `switch` can now be an expression and destructure records/objects with
   patterns, enabling concise, exhaustive branching.
